# Локальная LLM для Help Prompt: как «учить» и какую использовать

Цель проекта (по плану): сбор данных пользователя → шаблон промпта → вызов LLM → ответ в чате. Ниже — что значит «учить» в нашем случае, какую модель ставить на сервер и когда имеет смысл настоящее дообучение.

---

## 1. Что значит «учить» LLM в нашем контексте

В Help Prompt мы **не переобучаем веса модели** (fine-tune) на первом этапе. «Обучение» идёт за счёт:

| Способ | Описание |
|--------|----------|
| **Системный промпт** | Роль ассистента, правила (краткость, формат, темы), запреты. Модель ведёт себя согласно инструкции. |
| **Контекст из собранных данных** | Поля из CollectedData (имя, роль, цель и т.д.) подставляются в шаблон промпта — модель «знает» о пользователе. |
| **История диалога** | Последние N сообщений уходят в запрос к LLM — модель опирается на текущий разговор. |
| **Few-shot в промпте** | 1–2 примера «вопрос → желаемый ответ» в системном промпте улучшают формат и тон. |
| **RAG (позже)** | Если появится база документов/инструкций — релевантные куски подмешиваются в контекст. |

Этого достаточно, чтобы чат вёл сценарий сбора данных и давал релевантные ответы. Настоящий **fine-tune** (дообучение весов на своих данных) рассматриваем отдельно — см. раздел 4.

## 2. Параметры сервера и выбор модели

Текущий деплой — **VPS Beget** (Ubuntu). Типичные конфигурации:

| Тариф Beget | RAM | CPU | Что реально для Ollama |
|-------------|-----|-----|-------------------------|
| Минимальный | **2 GB** | 1 ядро | Только очень лёгкие модели (~1B параметров). Полная llama3.2 не влезет. |
| Рекомендуемый для чата | **4 GB** | 2 ядра | Модели 2–3B: приемлемое качество для сценариев и коротких ответов. |

**Важно:** на VPS обычно **нет GPU** — всё на CPU. Скорость генерации будет умеренной (несколько секунд на ответ для 1–3B).

### Рекомендации по моделям (Ollama)

| RAM | Модель | Размер | Комментарий |
|-----|--------|--------|-------------|
| **2 GB** | `llama3.2:1b` | ~1.3 GB | Уже упоминалась в DEPLOY-BEGET. Быстрая, простая. |
| **2 GB** | `phi3:mini` | ~2.3 GB | Может быть впритык; пробовать при 2 GB. |
| **2 GB** | `tinyllama` | ~0.6 GB | Очень быстрая, качество ниже. |
| **4 GB** | `llama3.2:3b` | ~2 GB | Хороший баланс скорость/качество для сценариев. |
| **4 GB** | `qwen2.5:3b` | ~2 GB | Хорошо следует инструкциям, подходит под «помощник». |
| **4 GB** | `phi3:medium` | ~7 GB | Не влезет в 4 GB. |
| **24+ GB** | Модели 7B–35B | 4–24 GB | Только если появится сервер/ПК с большим объёмом RAM или GPU (см. EVAL-Huihui-Qwen3.5-35B-chatbot.md). |

**Для нашей цели** (сбор данных, короткие ответы, следование сценарию):

- При **2 GB RAM** — **`llama3.2:1b`** или **`phi3:mini`** (если влезет).
- При **4 GB RAM** — **`qwen2.5:3b`** или **`llama3.2:3b`**: лучше качество и следование системному промпту при приемлемой скорости.

Команды на сервере:

```bash
# 2 GB RAM
ollama pull llama3.2:1b

# 4 GB RAM (выбрать одну)
ollama pull qwen2.5:3b
# или
ollama pull llama3.2:3b
```

В `.env` приложения: `OLLAMA_MODEL=qwen2.5:3b` (или выбранная модель).

## 3. Как «учить» без fine-tune: практические шаги

1. **Сервис промптов** (по плану): шаблоны с плейсхолдерами `{{field}}`, подстановка из CollectedData и контекста сессии.
2. **Системный промпт** задаёт роль, например:
   - «Ты помощник в сборе данных для персональных промптов. Задаёшь короткие уточняющие вопросы по сценарию. Отвечаешь кратко.»
3. **В запрос к LLM** передавать:
   - `system`: собранный системный промпт (шаблон + данные пользователя).
   - `messages`: последние 10–20 пар user/assistant из текущей сессии.
4. **Few-shot**: в системный промпт добавить 1–2 примера диалога «вопрос бота → ответ пользователя → следующий вопрос бота», чтобы зафиксировать формат.
5. **RAG** (когда появятся документы): хранить чанки в векторной БД, по запросу подставлять топ-N релевантных в контекст.

Так мы «учим» модель поведению и контексту без изменения весов.

## 4. Когда и как делать настоящее дообучение (fine-tune)

**Когда имеет смысл:**

- Нужен специфичный стиль, терминология или формат, которые плохо достигаются промптом.
- Есть много размеченных диалогов (сотни/тысячи) и задача «как в этих примерах».

**Ограничения по железу:**

- Fine-tune даже с LoRA/QLoRA для моделей 3B+ обычно требует **8+ GB RAM** или **GPU с 6+ GB VRAM**.
- На Beget (2–4 GB, без GPU) **полноценный fine-tune локально нереалистичен**.

**Варианты:**

| Вариант | Где | Плюсы / минусы |
|---------|-----|-----------------|
| **Облачный fine-tune** | OpenAI, Together, Modal, RunPod и т.д. | Не нагружаем свой сервер; платно; модель потом можно экспортировать или вызывать по API. |
| **Свой сервер / ПК** | 24+ GB RAM или GPU 8+ GB | Полный контроль; нужны данные и время на обучение. |
| **Ollama + Modelfile** | Локально | Не полноценный fine-tune, а кастомизация системного промпта и параметров при создании модели-образа — быстро и без больших ресурсов. |

**Практический вывод:** для текущей цели Help Prompt достаточно **хорошей базовой модели (2–3B) + продуманный системный промпт и шаблоны**. Fine-tune имеет смысл планировать только при появлении явной потребности и ресурсов (данные + более мощное железо или облако).

## 5. Краткий чеклист

- [ ] Уточнить объём RAM на текущем VPS (2 или 4 GB).
- [ ] Установить Ollama на сервер (см. DEPLOY-BEGET.md).
- [ ] Выбрать модель: при 2 GB — `llama3.2:1b`, при 4 GB — `qwen2.5:3b` или `llama3.2:3b`.
- [ ] В Help Prompt: реализовать сервис промптов (шаблоны + подстановка CollectedData).
- [ ] В запрос к Ollama передавать системный промпт + последние сообщения сессии.
- [ ] При необходимости добавить few-shot примеры в системный промпт.
- [ ] Fine-tune оставить на потом, когда будут данные и/или более мощное окружение.
