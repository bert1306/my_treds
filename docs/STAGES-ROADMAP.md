## Этап 0. Подготовка и база — ✅ Готово

**Цель:** проект поднят, БД и базовые сущности есть, инфраструктура для чата готова.

**Сделано:**
- **Проект:** Next.js (App Router, TypeScript, Tailwind) в `help-prompt/`
- **БД:** Prisma + SQLite; модели **User** (id, email?, name?), **Session** (id, userId, channel, title?, isFavorite, deviceId?), **Message** (id, sessionId, role, content, createdAt)
- **Session:** поле `title` (название диалога), `isFavorite`, `deviceId` — привязка к устройству для гостевого режима; `channel` (по умолчанию "web")
- **Роли сообщений:** в БД хранятся как `USER` / `ASSISTANT`; при выдаче в API приводятся к lowercase
- **Авторизация:** не реализована. Для новых сессий используется один общий пользователь: `User.findFirst()` или создаётся пользователь с `name: "Guest"`. Привязка пользователя к deviceId (один User на устройство) — позже; авторизация по email/Telegram — в перспективе.

**Результат:** репозиторий и приложение поднимаются, миграции применены, модели готовы к использованию в API чата. Сам чат (API и UI) реализован в рамках этапа 1.

## Этап 1. Веб-чат MVP — ✅ Готово

**Цель:** минимальный диалог в вебе с сохранением истории; список диалогов по устройству и управление ими.

**Сделано:**

**API:**
- **POST /api/chat** — тело: `{ message, deviceId?, sessionId? }`. Если `sessionId` нет: создаётся User (если в БД никого нет — с именем "Guest") и новая Session с этим userId, `channel: "web"`, переданным `deviceId` и `title` из первых 57 символов сообщения. Сообщение пользователя сохраняется в Message (role: USER), ответ-заглушка — в Message (role: ASSISTANT). В ответе возвращаются `reply` и `sessionId`. Если у существующей сессии не было deviceId — при передаче deviceId в теле запроса сессия обновляется (проставляется deviceId).
- **GET /api/sessions?deviceId=** — список сессий с данным deviceId (orderBy: isFavorite desc, createdAt desc); поля id, title, isFavorite, createdAt.
- **GET /api/sessions/[id]/messages** — сообщения выбранной сессии (без проверки владельца по deviceId — ограничение MVP).
- **PATCH /api/sessions/[id]** — обновление `title` и/или `isFavorite` (без проверки владельца по deviceId).
- **DELETE /api/sessions/[id]** — удаление сессии (без проверки владельца по deviceId).

**Фронт:**
- Экран чата: поле ввода, отправка, список сообщений (user/assistant), скролл к последнему сообщению; при первой отправке без выбранного диалога уходит запрос без sessionId, после ответа выставляется sessionId и обновляется список диалогов.
- Список диалогов: боковая панель (десктоп) и оверлей (мобильный); загрузка по deviceId из localStorage (`help-prompt-device-id`); при выборе диалога — загрузка сообщений через GET sessions/[id]/messages.
- Управление диалогами: переименование (PATCH title), избранное (PATCH isFavorite), удаление (DELETE); кнопка «Новый диалог» — сброс sessionId и очистка экрана.
- Заглушка ответа: фиксированный текст вида «Вы написали: «…». Это заглушка — позже здесь будет ответ ИИ.» (без вызова LLM).

**Ограничения (принять к сведению, доработать при необходимости):**
- Владелец сессии не проверяется: PATCH/DELETE и GET messages по id сессии не сверяют deviceId — подходит для одного устройства/доверенного окружения; при мультитенанте или публичном доступе нужна проверка на этапе 7 или при появлении авторизации.
- Один общий User для всех гостевых сессий (или первый попавшийся User) — список «моих» диалогов определяется только по deviceId в Session.

**Результат:** пользователь открывает веб-чат, ведёт диалог, сообщения сохраняются; видит список своих диалогов по deviceId, может переименовывать, помечать избранным и удалять; готовность к подключению сценариев (этап 2) и ИИ (этап 4).

## Этап 2. Сценарии и сбор данных (мастер) — ✅ Готово

**Цель:** бот задаёт вопросы по сценарию (мастер из ТЗ); ответы сохраняются в структурированном виде и используются для выбора конструктора/шаблона.

**Сделано:**
- **Prisma:** модель **CollectedData** (sessionId, key, value), уникальный индекс (sessionId, key). Модель Scenario не выносилась в БД — шаги мастера заданы в коде (`src/lib/wizard.ts`).
- **Шаги сценария:** тип choice (варианты выбора) или text; ключ поля в CollectedData (goal, goalDetail, role, detailLevel, context); для шага 2 варианты детализации зависят от выбора цели (динамические опции в коде).
- **Мастер из ТЗ:** 5 шагов — цель → детализация цели → роль → уровень детализации ответа → контекст (опционально, свободный текст с кнопкой «Пропустить»).
- **Бэкенд:** при запросе текущего шага загружаются CollectedData по sessionId; функция `getCurrentStep(collected)` возвращает следующий незаполненный шаг или признак завершения; при POST ответа — upsert в CollectedData, возврат следующего шага или `completed: true`.
- **API:** POST /api/sessions (body: deviceId) — создание пустой сессии для мастера; GET /api/sessions/[id]/wizard/step — текущий шаг или { completed: true }; POST /api/sessions/[id]/wizard/step (body: { value }) — отправить ответ, получить следующий шаг или completed.
- **Фронт:** кнопка «Настроить параметры (мастер)» на приветственном экране и «+ Новый диалог» в боковой панели создают пустую сессию и показывают мастер; отображение вопроса, кнопки выбора или поле ввода + «Далее»/«Пропустить»; после прохождения — переход к обычному чату (приветственный экран с полем ввода). Кнопка «Отменить» в мастере возвращает в главное меню.

**Перед первым запуском:** выполнить миграцию БД: `cd help-prompt && npx prisma migrate deploy` (или `npx prisma migrate dev`).

**Результат:** пользователь может пройти мастер из 5 шагов перед чатом; данные сохраняются в CollectedData по сессии; готовность к этапу 3 (шаблоны и выбор конструктора).

**Зависимости:** этап 0–1.  
**Следующий этап:** 3 (шаблоны и сервис генерации промптов).

## Этап 3. Шаблоны промптов и сервис генерации

**Цель:** по собранным данным и выбранному конструктору/шаблону формировать итоговый промпт (системный и/или пользовательский); заложить правила выбора шаблона по цели/роли/типу.

**Критерии готовности:**
- [ ] В Prisma добавлена модель **PromptTemplate**: name, type (system / user), body с плейсхолдерами `{{field}}`, категория/теги (для привязки к мастеру), опционально — правила (цель, роль, тип задачи).
- [ ] Сервис генерации промптов: на вход — sessionId (или userId) + идентификатор шаблона (или авто-выбор по правилам); загрузка CollectedData по сессии/пользователю; подстановка в шаблон → итоговый текст промпта.
- [ ] Правила выбора конструктора (как в ТЗ): например «ЕСЛИ цель = "Создать контент" И тип = "Статья" И роль = "Маркетолог" → шаблон "SEO-статья"»; реализация в коде или в данных (таблица правил / конфиг).
- [ ] Минимум 2–3 рабочих шаблона под MVP-категории ТЗ (например: «Объясни простыми словами», «Напиши статью», «Опиши тренды») с плейсхолдерами под поля мастера.
- [ ] API: список доступных шаблонов (по сессии/CollectedData); получение сгенерированного промпта по выбранному шаблону (для предпросмотра и для отправки в LLM на этапе 4).

**Результат:** по данным мастера можно автоматически или вручную выбрать шаблон и получить готовый промпт; сервис промптов — единая точка для веба и позже для Telegram.

**Зависимости:** этап 2.  
**Следующий этап:** 4 (подключение LLM).

## Этап 4. Подключение LLM

**Цель:** ответы в чате генерирует ИИ; промпт собирается из шаблона и CollectedData; поддержка локальной модели (Ollama) и опционально облачного API.

**Критерии готовности:**
- [ ] Интеграция с Ollama: вызов API (или локальный сервис) с системным промптом + последние N сообщений диалога; сохранение ответа в `Message` и отдача клиенту.
- [ ] Выбор модели по конфигу/ресурсам (см. `docs/LLM-LOCAL-LEARNING.md`); при недоступности Ollama — понятная ошибка или fallback (опционально — OpenAI API).
- [ ] В чате: при отправке сообщения пользователем — сборка системного промпта (из шаблона + CollectedData), добавление истории сообщений, запрос к LLM, сохранение ответа assistant в Message.
- [ ] Параметры генерации (temperature, max_tokens и т.д.) — в конфиге или в настройках сессии/профиля; передача в API LLM (без жаргона в UI по ТЗ: «точность ответа», «развёрнутость» при необходимости позже).
- [ ] Обработка ошибок (таймаут, недоступность модели, превышение лимита) и отдача пользователю понятного сообщения.

**Результат:** полноценный диалог с ИИ; промпт строится из мастера и шаблонов; можно тестировать сценарии и шаблоны.

**Зависимости:** этап 3.  
**Следующий этап:** 5 (улучшение веба и экспорт).

## Этап 5. Улучшение веб-чата и экспорт

**Цель:** стабильный UX, стриминг ответа (если API поддерживает), экспорт/копирование промпта по ТЗ.

**Критерии готовности:**
- [ ] Индикация «печатает» (typing) во время запроса к LLM.
- [ ] Стриминг ответа от LLM (если выбранный провайдер поддерживает stream); посимвольное или по токенам отображение в чате.
- [ ] Экспорт готового промпта: копирование в буфер обмена; опционально — сохранение в файл (.md / .txt) по ТЗ.
- [ ] При необходимости: кнопка «Скопировать промпт» для текущего контекста (системный промпт или последний сгенерированный пользовательский).
- [ ] История сессий уже есть (этап 0–1); при необходимости — доработка отображения и навигации.

**Результат:** удобный веб-чат с обратной связью и возможностью переносить промпт во внешние системы (ChatGPT, Claude и т.д.).

**Зависимости:** этап 4.  
**Следующий этап:** 6 (Telegram).

## Этап 6. Telegram

**Цель:** тот же логический поток (мастер → шаблоны → промпт → LLM) в Telegram-боте; один бэкенд для веба и Telegram.

**Критерии готовности:**
- [ ] Бот создан (BotFather), токен в конфиге; сервис на Telegraf или node-telegram-bot-api.
- [ ] Идентификация по `telegram_id`: создание/поиск User или привязка к существующей сущности (Session с channel=telegram).
- [ ] Один слой бизнес-логики: создание сессии (channel=telegram), сохранение Message, запуск сценария мастера, генерация промпта, вызов LLM — переиспользование кода из веба.
- [ ] Адаптация мастера под Telegram: короткие сообщения, кнопки (reply keyboard / inline) для выбора цели, роли, уровня детализации и т.д.
- [ ] Ответы ИИ в Telegram (текст; при длинном ответе — разбивка или файл по необходимости).
- [ ] Деплой: процесс бота (PM2 или отдельный контейнер), доступ к той же БД и сервису промптов.

**Результат:** пользователь может пройти мастер и получить ответ ИИ и в вебе, и в Telegram; данные и шаблоны общие.

**Зависимости:** этапы 2, 3, 4.  
**Следующий этап:** 7 (безопасность и эксплуатация).

## Этап 7. Безопасность и эксплуатация

**Цель:** ограничение злоупотреблений, политики данных, наблюдаемость.

**Критерии готовности:**
- [ ] Rate limit на API (чат, создание сессий) и при необходимости на запросы к LLM.
- [ ] Политика конфиденциальности и согласие на сбор данных (где применимо); хранение только необходимых полей.
- [ ] Логирование без тел сообщений в продакшене (или только с явным согласием); аудит доступа при необходимости.
- [ ] Резервное копирование БД; мониторинг (здоровье API, ошибки LLM, алерты по необходимости).

**Результат:** приложение готово к использованию с учётом безопасности и эксплуатации.

**Зависимости:** после этапов 5–6.  
**Дальше:** пост-MVP по ТЗ (подсказки по формулировкам, тон и стиль, генерация системного промпта для внешнего ИИ, расширенные категории).
