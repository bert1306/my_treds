## 1. Что из ТЗ извлечено

### 1.1 Продукт и цель

- **Название по ТЗ:** PromptCraft — приложение-конструктор промптов с локальной ИИ.
- **Суть:** помощь пользователю в создании эффективных промптов для LLM (ChatGPT, Claude, Gemini и др.) через пошаговый сценарий и шаблоны.
- **«Craft»:** именно конструирование/создание промптов, а не только чат с ИИ.

Цель продукта совпадает с уже описанным в `docs/PLAN-WEB-CHAT-AI-PROMPTS.md`: сбор данных о пользователе → подстановка в шаблоны → формирование промпта → вызов LLM.

### 1.2 Целевые платформы по ТЗ

| Платформа | В ТЗ | Наше решение |
|-----------|------|----------------|
| **Web** | Да | Основной интерфейс (уже есть Help Prompt). |
| **Telegram** | Да | Второй канал (по плану — после веба). |
| **Десктоп (Windows/macOS/Linux)** | Tauri / Electron | Пока не делаем. |
| **Мобильные (iOS/Android)** | Отдельная фаза, свои ограничения | Пока не делаем. |

Оставляем только **Web** и **Telegram**. Нативные и десктоп — в перспективе; если пойдём в сторону мини-апп или нативных приложений, имеет смысл рассматривать **React** (общая кодовая база с Next.js) для переиспользования UI-логики и состояния.

### 1.3 Локальная ИИ и модели

В ТЗ упоминаются:

- **Локальные модели:** Llama 3.1 (8B, 70B), Mistral 7B, Gemma 2, Qwen 2.5; требования по RAM; квантизация (INT8, INT4 и т.д.); при недоступности — graceful degradation.
- **Облако:** API ChatGPT/OpenAI как опция или fallback.

Мы уже зафиксировали в `docs/LLM-LOCAL-LEARNING.md`: Ollama, выбор модели под ресурсы сервера (2–4 GB → лёгкие модели; 24+ GB — при наличии), при необходимости — облачный API.

### 1.4 Функциональность из ТЗ (MVP и дальше)

- **Онбординг / вход:** веб (логин, email, OAuth и т.д.) и Telegram (по telegram_id).
- **Мастер (wizard):** пошаговый сбор параметров: цель, роль, тип задачи, уровень детализации и т.д. → сохранение в «профиль» пользователя.
- **Выбор «конструктора» по правилам:** в зависимости от цели/роли/типа подбирается тип промпта (например: SEO-статья, анализ данных, объяснение, код, HR-тексты и т.д.).
- **Шаблоны промптов:** системный и пользовательский промпты с плейсхолдерами; подстановка собранных данных.
- **Параметры генерации:** temperature, max_tokens, top_p и т.д. (в т.ч. для совместимости с облачными API).
- **Экспорт/копирование:** готовый промпт — в буфер, .md/.txt, при необходимости интеграция с внешним API или QR.

Часть этого уже заложена в нашем плане (сценарии, CollectedData, PromptTemplate, вызов LLM); остальное — уточнение и приоритизация.

### 1.5 Пользовательские сценарии (User Flow)

- **Веб:** вход → (онбординг) → мастер за N шагов → выбор/настройка конструктора → генерация промпта → ответ LLM → копирование/экспорт.
- **Telegram:** старт бота → те же логические шаги в формате диалога (короткие сообщения, кнопки).

Целевые метрики из ТЗ (ориентир): время до первого результата, retention, NPS и т.д. — можно вынести в отдельный бэклог после MVP.

### 1.6 Детали из полного ТЗ (Word)

**Мастер (навигационный подбор конструктора) — 5 шагов:**

1. **Цель обращения к ИИ:** Получить информацию / Создать контент / Проанализировать данные / Проверить или улучшить / Другое → базовая категория конструктора.
2. **Детализация цели:** динамический список на основе шага 1.
3. **Роль пользователя:** Разработчик / Продакт / Аналитик / Студент / Маркетолог / Учитель / Другое / Не указывать → адаптация лексики и предустановок.
4. **Уровень детализации ответа:** Краткий обзор / Подробное объяснение / Пошаговая инструкция / Сравнение вариантов.
5. **Контекст (опционально):** свободный текст, данные или описание ситуации.

Правила выбора конструктора — пример: «ЕСЛИ цель = "Создать контент" И тип = "Статья" И роль = "Маркетолог" → конструктор = "SEO-оптимизированная статья"».

**MVP — три категории задач:**

- **Написание текстов:** маркетинговый контент, деловая коммуникация, креативное письмо, редактирование/улучшение, SEO-контент (подкатегории с параметрами конструктора).
- **Обучение:** объяснение концепций, структурирование обучения, проверка знаний, практическая помощь, подготовка к экзаменам.
- **Анализ данных:** описательная/диагностическая/предиктивная аналитика, презентация результатов (с оговорками об ограничениях и неопределённости).

**Библиотека шаблонов:** иерархия по категориям и ролям; прогрессивное раскрытие; метаданные (название, описание, теги, сложность, estimated time); примеры заполнения; рейтинги и курирование.

**Подсказки по формулировкам:** три уровня — информационный (при начале ввода), предупреждающий (расплывчатая формулировка), корректирующий (после ввода — рерайт с объяснением). Детектор расплывчатости: отсутствие контекста, неопределённые прилагательные, отсутствие формата вывода, противоречивые требования.

**Настройка тона и стиля:** шкала формальности (академический ↔ разговорный); параметры — экспертность, детализация, структурированность, креативность; предустановленные профили (Деловой, Дружелюбный, Экспертный, Менторский, Креативный). В интерфейсе — без жаргона («точность ответа» вместо «temperature», «развёрнутость» вместо «max_tokens»).

**Помощник настройки внешнего ИИ:** генерация **системного промпта** для ChatGPT/Claude/Gemini по опросу (цели использования, стиль общения, границы применимости, формат ответов). Экспорт: буфер обмена, .md/.txt, API, QR-код.

**Быстрый путь (User Flow):** ввод в поле быстрого старта → автовыбор конструктора → (опционально) уточнение → готовый промпт → копирование/сохранение; целевое время &lt; 30 сек. **Продвинутый путь:** категория → роль → шаблон/конструктор → настройка параметров → редактирование → финализация.

**Качество (ТЗ):** понятность интерфейса (80% создают первый промпт за &lt; 5 мин), корректность промптов (&gt; 85% релевантные ответы), стабильность локальных моделей, офлайн-доступность базового функционала; метрики — время до промпта, retention D7/D30, NPS, кол-во промптов на пользователя.

**Язык:** интерфейс и контент — только русский (стратегия для РФ рынка); архитектура допускает последующую интернационализацию.

## 2. Соответствие нашему текущему плану

Наш план (`PLAN-WEB-CHAT-AI-PROMPTS.md`) и уже реализованный **Help Prompt** хорошо стыкуются с ТЗ:

| ТЗ | Наш план / факт |
|----|------------------|
| Веб-чат | Help Prompt: чат, сессии, сообщения, список диалогов. |
| Сбор данных пользователя | Этап 2: Scenario, CollectedData, пошаговые вопросы. |
| Шаблоны промптов | Этап 3: PromptTemplate, плейсхолдеры `{{field}}`. |
| Вызов LLM | Этап 3: Ollama/OpenAI, системный промпт + история. |
| Telegram | Этап 5: тот же бэкенд, channel=telegram. |
| Локальная ИИ | Ollama + выбор модели под железо (см. LLM-LOCAL-LEARNING.md). |

Отличия/дополнения по ТЗ:

- Явный **мастер (wizard)** с выбором цели/роли/типа задачи и правилами выбора «конструктора» — у нас это можно реализовать как сценарии (Scenario) и привязку шаблонов к комбинациям полей.
- **Экспорт промпта** (копирование, .md/.txt) — отдельная фича в веб-чате и при желании в Telegram.
- **Настройки генерации** (temperature, max_tokens и т.д.) — хранить в конфиге или в сессии/профиле и передавать в API LLM.

## 3. Что планируется сделать (без кода, только план)

### 3.1 Интерфейсы

- **Web:** единственный UI на первом этапе; уже есть (Help Prompt). Доращивать: сценарии сбора, шаблоны промптов, вызов локальной/облачной LLM, при необходимости — экспорт промпта.
- **Telegram:** второй интерфейс после стабилизации веба; тот же бэкенд, те же сценарии и шаблоны; адаптация подачи (короткие шаги, кнопки).
- **Нативные приложения:** пока не планируем. Если позже появятся мини-апп или нативные клиенты — целесообразно вынести общую логику (состояние, типы, сценарии) в переиспользуемый слой; UI на React может быть основой для веба и для встраивания в другие оболочки.

### 3.2 Фазы разработки (соответствие этапам нашего плана)

1. **Веб-чат и база (уже частично есть)**  
   Сессии, сообщения, диалоги, привязка к устройству/гостю. При необходимости — явная модель пользователя (email/telegram_id позже).

2. **Сценарии и сбор данных (CollectedData + Scenario)**  
   - Описание сценария: шаги (вопрос, выбор, свободный текст), порядок, привязка ответа к ключу в CollectedData.  
   - В чате: бот задаёт вопросы по сценарию; ответы пишутся в CollectedData; по завершении сценария — переход к «обычному» чату с LLM.  
   - При желании — аналог «мастера» из ТЗ: первый вход или отдельная команда «Настроить профиль» запускает сценарий (цель, роль, тип задач и т.д.).

3. **Шаблоны промптов и сервис генерации**  
   - PromptTemplate: системный/пользовательский, тело с `{{field}}`.  
   - Сервис: по user/session и идентификатору шаблона (или правилу «цель+роль+тип») подставлять CollectedData → итоговый промпт.  
   - Правила выбора шаблона (как «конструктор» в ТЗ): по полям из CollectedData выбирать, какой шаблон применять.

4. **Подключение LLM**  
   - Локально: Ollama, модель по конфигу/железу (см. LLM-LOCAL-LEARNING.md).  
   - Опционально: облачный API (OpenAI и др.) как fallback или альтернатива.  
   - В чате: при отправке сообщения — сборка промпта (системный + при необходимости пользовательский), последние N сообщений, вызов LLM, сохранение ответа в Message.

5. **Улучшение веб-чата**  
   - Стриминг ответа (если API поддерживает).  
   - Индикация «печатает», обработка ошибок.  
   - Экспорт/копирование готового промпта (по ТЗ).

6. **Telegram**  
   - Бот (Telegraf / node-telegram-bot-api), идентификация по telegram_id, те же сценарии и сервис промптов, адаптация подачи под Telegram.

7. **Безопасность и эксплуатация**  
   - Rate limit, политика данных, логирование без тел сообщений при необходимости.

### 3.3 Реакт и возможные мини-апп / нативные приложения

- Текущий веб уже на **React (Next.js)**. Всю продуктовую логику (сценарии, сбор данных, генерация промпта, вызов LLM) целесообразно держать на бэкенде (API), а фронт — тонким клиентом.  
- Если позже появятся:  
  - **Telegram Mini App** — можно сделать на том же React (или облегчённый SPA), вызывающий те же API.  
  - **Нативные приложения (iOS/Android)** — либо WebView с тем же веб-приложением, либо отдельный нативный UI с теми же API; общий React-код не обязателен, но типы и контракты API будут общими.  
- Имеет смысл не завязывать фронт на детали Next.js (где не нужно), чтобы при необходимости вынести общие компоненты в отдельную библиотеку или переиспользовать в мини-апп.
