# Этап 3: шаблоны промптов и сервис генерации — пошаговый план

Детализация этапа 3 из `STAGES-ROADMAP.md`. Результат: по данным мастера (и при необходимости выбору пользователя) формируется готовый текст промпта; сервис — единая точка для веба и этапа 4 (LLM).

Шаблоны под кнопки описаны в `docs/PROMPT-TEMPLATES-MVP.md`.

---

## Шаг 3.1. Модель PromptTemplate в Prisma

**Цель:** хранить шаблоны в БД (или иметь возможность подгружать из конфига/кода; для MVP допустимо зашить 4 шаблона в код и не заводить таблицу сразу — см. альтернативу ниже).

**Вариант A — таблица в БД**

- Добавить модель **PromptTemplate**:
  - `id` (cuid)
  - `slug` (string, unique) — идентификатор для API и привязки к preset (trends, formulate, ideas, summary)
  - `name` (string) — человекочитаемое название
  - `type` (string) — `"user"` или `"system"` (для MVP достаточно user)
  - `body` (string) — тело шаблона с плейсхолдерами `{{context}}`, `{{detailLevel}}`, `{{role}}` и т.д.
  - `goal` (string, optional) — для правил авто-выбора (analyze, info, content, improve)
  - `goalDetail` (string, optional) — trends, explain, creative, resize
  - `createdAt`, `updatedAt`
- Миграция.

**Вариант B — только в коде (быстрее для MVP)**

- Не заводить таблицу; хранить массив из 4 шаблонов в `lib/prompt-templates.ts` (или рядом с wizard).
- Поля: slug, name, type, body. Правила выбора «по goal + goalDetail» — в коде (map по preset id).
- Плюс: нет миграций, быстрый старт. Минус: смена шаблона только через деплой. Позже можно перенести в БД.

**Рекомендация:** начать с варианта B (4 шаблона в коде), после этапа 4 при необходимости вынести в БД.

**Критерий готовности:** есть 4 шаблона с body из PROMPT-TEMPLATES-MVP.md, доступ по slug (trends, formulate, ideas, summary).

## Шаг 3.2. Плейсхолдеры и подстановка

**Цель:** единая функция «подставить данные в шаблон → строка».

- **Вход:** тело шаблона (строка), объект данных (context, detailLevel, role, goal, goalDetail, name и т.д.).
- **Правила подстановки:**
  - `{{fieldName}}` → значение `data[fieldName]` или пустая строка.
  - Опционально: условный блок «если role задана и не none» — либо фраза «Учти мою роль: {{role}}», либо пропуск блока при пустом role. В первой версии можно без условных блоков: одна строка «Роль: {{role}}» и при пустом role не добавлять её в итог (или подставлять пустую строку и обрезать лишние переносы).
- **Маппинг detailLevel в человекочитаемый текст** (для вставки в промпт): brief → «краткий обзор», detailed → «подробное объяснение», stepwise → «пошаговая инструкция», compare → «сравнение вариантов». Хранить в коде (константа или функция).

**Файл:** `lib/prompt-service.ts` (или `lib/build-prompt.ts`) — функция `buildPromptFromTemplate(templateBody, data): string`.

**Критерий готовности:** по шаблону и объекту данных возвращается одна строка без `{{...}}` (все плейсхолдеры заменены).

## Шаг 3.3. Сбор данных для подстановки (сессия + профиль)

**Цель:** по sessionId собрать объект для подстановки: CollectedData мастера + при необходимости роль/имя из User.

- Загрузить по sessionId: все записи CollectedData (goal, goalDetail, role, detailLevel, context).
- Если в CollectedData нет role, но есть у User (сессия → user → role) — подставить из профиля.
- Опционально: подставить name из User для обращения.
- Результат: объект `{ context, detailLevel, role, goal, goalDetail, name?, ... }` с строковыми значениями (для role/detailLevel при необходимости подставить человекочитаемые подписи из маппинга).

**Файл:** в том же `lib/prompt-service.ts` — функция `getCollectedDataForSession(sessionId): Promise<Record<string, string>>`. Внутри: prisma.collectedData + при необходимости prisma.session с user.

**Критерий готовности:** по sessionId получаем плоский объект ключ–значение для подстановки в шаблон.

---

## Шаг 3.4. Выбор шаблона по preset / правилам

**Цель:** по данным сессии (или по явному выбору) определить, какой шаблон использовать.

- **Явный выбор:** API вызывается с параметром `templateSlug` или `presetId` (trends, formulate, ideas, summary) — берём шаблон с таким slug.
- **Авто-выбор (опционально для MVP):** по полям goal + goalDetail (и при желании role) выбирать slug из правил. Пример: goal=analyze, goalDetail=trends → trends; goal=info, goalDetail=explain → formulate; goal=content, goalDetail=creative → ideas; goal=improve, goalDetail=resize → summary. Реализовать как map в коде.

**Файл:** в `lib/prompt-service.ts` — функция `getTemplateByPresetId(presetId): Template` и при необходимости `selectTemplateSlugByCollectedData(collected): string`.

**Критерий готовности:** по preset id (или по collected) возвращается один выбранный шаблон (body + slug).

---

## Шаг 3.5. Сервис «сгенерировать промпт для сессии»

**Цель:** одна точка входа для веба и этапа 4.

- **Сигнатура:** `generatePromptForSession(sessionId: string, options: { templateSlug?: string; presetId?: string }): Promise<string>`.
- **Логика:**  
  1) Собрать данные для сессии (шаг 3.3).  
  2) Определить шаблон: по templateSlug или по presetId, при отсутствии — по авто-выбору из collected (шаг 3.4).  
  3) Подставить данные в body шаблона (шаг 3.2).  
  4) Вернуть итоговую строку.
- Если сессии нет или нет данных для выбора шаблона — бросить ошибку или вернуть дефолтный промпт (на усмотрение).

**Файл:** `lib/prompt-service.ts` — экспорт `generatePromptForSession`.

**Критерий готовности:** по sessionId (+ slug или presetId) возвращается готовый текст промпта.

## Шаг 3.6. API для фронта и этапа 4

**Цель:** дать фронту список шаблонов и сгенерированный промпт (для предпросмотра и для отправки в LLM).

1. **GET /api/sessions/[id]/prompt?deviceId=xxx&presetId=trends** (или `templateSlug=trends`)  
   - Проверка владельца сессии по deviceId (getSessionIfOwned).  
   - Вызов `generatePromptForSession(sessionId, { presetId })` (или templateSlug).  
   - Ответ: `{ prompt: string }` или 404/403.

2. **GET /api/prompt-templates** (опционально)  
   - Список доступных шаблонов: id/slug, name (для отображения в UI).  
   - Без привязки к сессии; можно без deviceId.  
   - Ответ: `{ templates: { slug, name }[] }`.

**Критерий готовности:** фронт может запросить сгенерированный промпт по sessionId + presetId (или slug); при необходимости — список шаблонов.

## Шаг 3.7. Интеграция во фронт (предпросмотр промпта)

**Цель:** после прохождения мастера пользователь может увидеть готовый промпт (и позже скопировать или отправить в чат).

- После завершения мастера (wizard completed) для текущей сессии известны goal, goalDetail, role, detailLevel, context.
- На экране чата (или перед первым сообщением) показать блок «Ваш промпт»: запрос GET к `/api/sessions/[id]/prompt?deviceId=...&presetId=...` (presetId можно вывести из collected: goal+goalDetail → preset, либо дать выбор из 4 пунктов).
- Отобразить текст промпта; кнопка «Скопировать» (этап 5 можно расширить).
- При отправке первого сообщения в чат на этапе 4 этот же промпт будет использоваться как контекст/системный или первое пользовательское сообщение (по решению этапа 4).

**Критерий готовности:** пользователь видит сгенерированный промпт после мастера; presetId определяется по данным сессии или выбору.
