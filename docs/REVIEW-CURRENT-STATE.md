# Критический обзор текущего состояния Help Prompt

Краткая оценка того, что уже сделано: что хорошо, что — слабые места или риски, что стоит поправить до этапа 3.

---

## 1. Что сделано хорошо

**Модель данных**
- Чёткое разделение: User (по deviceId), Session (диалог), Message, CollectedData. Связи и каскады настроены разумно.
- Профиль в User (имя, роль, заготовки под логин/пароль/telegram) — логично, один пользователь на устройство через deviceId.

**Мастер и пресеты**
- Пять шагов по ТЗ, динамическая детализация цели, опциональный контекст. Логика в `wizard.ts` читаемая, легко расширять.
- Топовые кнопки по роли (статический порядок) — понятно, потом можно заменить на данные из аналитики.
- Роль из профиля подставляется в мастере — шаг «роль» не дублируется, консистентно.

**API**
- GET sessions по deviceId, POST сессии и чата с getOrCreateUserByDeviceId — единая точка «кто пользователь».
- Профиль: GET/PATCH по deviceId, без пароля в ответах.

**Фронт**
- Один главный экран: приветствие, кнопки (по роли), мастер, чат. Профиль вынесен на отдельную страницу, в шапке есть ссылка.
- Имя в приветствии, подпись под кнопками в зависимости от роли — мелочи, но улучшают ощущение «под меня».

**Итог по плюсам:** Основа для этапа 3 (шаблоны и генерация промпта) заложена. Данные мастера и профиля уже можно использовать при сборке промпта.

## 2. Слабые места и риски

### 2.1 Нет проверки владельца сессии (критично при многопользовательском/публичном доступе)

**Суть:** Сейчас ни один API не проверяет, что сессия принадлежит тому, кто делает запрос.

- **GET /api/sessions/[id]/messages** — отдаёт сообщения по `sessionId` без проверки. Тот, кто знает или угадал id диалога, может прочитать чужие сообщения.
- **PATCH /api/sessions/[id]** — можно переименовать или поменять избранное у чужой сессии.
- **DELETE /api/sessions/[id]** — можно удалить чужой диалог.
- **POST /api/chat** — если в теле передать `sessionId` чужой сессии, сообщения допишутся в неё.

Список диалогов мы фильтруем по `deviceId`, поэтому в обычном сценарии пользователь видит только свои. Но идентификатор сессии можно подобрать (cuid не секретный) или получить с другого устройства/браузера. Для доверенной среды или одного пользователя это допустимо; для публичного или мультипользовательского — дыра.

**Сделано:** Введена проверка «сессия принадлежит устройству»:
- Хелпер `getSessionIfOwned(sessionId, deviceId)` в `lib/session.ts`: возвращает сессию, `not_found` (404) или `forbidden` (403).
- GET/PATCH/DELETE sessions/[id], GET messages, POST chat (при переданном sessionId), GET/POST wizard/step, POST wizard/preset — везде требуется deviceId и проверка владельца; при несовпадении — 403, при отсутствии сессии — 404.
- Фронт передаёт deviceId (query или body) во все эти запросы и при 403/404 вызывает goToMainMenu() (или сбрасывает состояние).

### 2.2 Профиль привязан только к deviceId

**Суть:** Кто знает deviceId — может читать и менять профиль (GET/PATCH с этим deviceId). deviceId хранится в localStorage и по идее не передаётся по сети явно в URL, но если он утёк или подбирается — контроль над профилем теряется.

**Оценка:** Для текущей стадии (без логина) приемлемо. При появлении авторизации профиль нужно будет привязать к аутентифицированному пользователю и не полагаться только на deviceId.

### 2.3 Chat без deviceId

**Суть:** В POST /api/chat при отсутствии `deviceId` вызывается `getOrCreateUserByDeviceId("")`. Создаётся/находится один пользователь с `deviceId: ""`. Все такие «анонимные» сессии привязаны к одному пользователю.

**Оценка:** Краевой случай (фронт сейчас всегда передаёт deviceId). Имеет смысл либо явно требовать deviceId в чате, либо оставить поведение, но зафиксировать в коде/доках, что пустой deviceId — один общий «гостевой» пользователь.

### 2.4 Нет 404 для несуществующей сессии

**Суть:** GET/POST wizard/step и GET messages для несуществующего `sessionId` отрабатывают «тихо»: wizard отдаёт первый шаг, messages — пустой массив. Пользователь может не понять, что диалог удалён или id неверный.

**Сделано:** При отсутствии сессии API возвращает 404; при «чужой» сессии — 403. Фронт при 403/404 сбрасывает состояние (goToMainMenu / обновление списка диалогов).

### 2.5 Ответ чата — заглушка

**Суть:** Ответ всегда текст «Вы написали: … Это заглушка — позже здесь будет ответ ИИ.» Ожидаемо до этапа 4 (LLM), но с точки зрения продукта диалог пока не «живой».

**Оценка:** Не недостаток, а запланированное состояние. Уйдёт с подключением LLM и шаблонов.

## 3. Что сделано по рекомендациям

1. **Проверка владельца сессии по deviceId** — реализована (см. п. 2.1 выше). Все операции с сессией требуют deviceId и проверяют владельца.

2. **404 при отсутствии сессии** — реализовано: getSessionIfOwned возвращает not_found → 404, фронт обрабатывает.

3. **Оставлено как есть (зафиксировано)**  
   - Привязка профиля к deviceId до появления логина.  
   - Поведение при пустом deviceId в чате (один общий пользователь).  
   - Статический порядок топовых пресетов по роли с возможностью позже заменить на данные из аналитики.

## 4. Итог

- **Архитектура и данные** — в порядке, готовы к этапу 3 (шаблоны и генерация промпта).
- **Главный риск** — отсутствие проверки владельца сессии; имеет смысл закрыть до выхода в публичный/мультипользовательский режим.
- Остальное — точечные улучшения (404, явное требование deviceId или документирование краевых случаев).
