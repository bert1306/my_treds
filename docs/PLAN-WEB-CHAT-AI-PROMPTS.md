# План разработки веб-чата для сбора данных и генерации промптов под ИИ

Цель: веб-чат, который собирает и обрабатывает данные о пользователях для формирования промптов к ИИ; затем — интеграция в Telegram.

---

## 1. Цели и границы проекта

| Цель | Описание |
|------|-----------|
| **Веб-чат** | Интерфейс для диалога, где пользователь общается с ботом/ИИ. |
| **Сбор данных** | Фиксация ответов, контекста, предпочтений пользователя для персональных промптов. |
| **Обработка данных** | Агрегация, нормализация, шаблонизация → готовый промпт для выбранной модели. |
| **Генерация промптов** | На основе собранных данных формируется системный/пользовательский промпт и отправляется в ИИ. |
| **Telegram** | После веб-версии — тот же логический поток (сбор → обработка → промпт → ИИ) в Telegram-боте. |

---

## 2. Рекомендуемый технологический стек

### 2.1 Фронтенд (веб-чат)

| Технология | Назначение |
|------------|------------|
| **Next.js 14+ (App Router)** | SPA/SSR, API routes, авторизация, удобный деплой. |
| **React + TypeScript** | UI чата, формы, состояние. |
| **Tailwind CSS** | Стили, адаптив. |
| **React Query / SWR** | Кэш и запросы к API. |
| **Zustand / React Context** | Глобальное состояние (профиль, сессия чата). |

### 2.2 Бэкенд и хранение

| Технология | Назначение |
|------------|------------|
| **Next.js API Routes / отдельный Node-сервис** | REST или tRPC для чата, профилей, сценариев. |
| **PostgreSQL или SQLite (Prisma)** | Пользователи, сессии, сообщения, собранные данные, шаблоны промптов. |
| **Redis (опционально)** | Кэш сессий, очереди, rate limit. |

### 2.3 ИИ и промпты

| Технология | Назначение |
|------------|------------|
| **Ollama / OpenAI API / иной провайдер** | Вызов LLM по сгенерированному промпту. |
| **Единый слой (сервис промптов)** | Сборка системного и пользовательского промпта из шаблонов и данных. |

### 2.4 Telegram (этап 2)

| Технология | Назначение |
|------------|------------|
| **node-telegram-bot-api / Telegraf** | Приём сообщений, отправка ответов. |
| **Общий бэкенд** | Та же БД и сервис промптов, что и для веба. |

## 4. Модель данных (ядро)

### 4.1 Основные сущности

- **User** — id, email/telegram_id, создан, обновлён.
- **Session** — сессия чата (привязка к user, канал: web / telegram).
- **Message** — сообщение в сессии (role: user/assistant, content, timestamp).
- **CollectedData** — ключ-значение или JSON по пользователю (ответы на вопросы, предпочтения, демография и т.д.).
- **PromptTemplate** — шаблон промпта (системный/пользовательский) с плейсхолдерами `{{field}}`.
- **Scenario** — сценарий сбора (какие вопросы задавать, в каком порядке, куда писать в CollectedData).

Связи: User → Session → Message; User → CollectedData; PromptTemplate использует поля из CollectedData.

### 4.2 Пример использования

1. Пользователь заходит в веб-чат.
2. Бот задаёт вопросы по сценарию (имя, роль, цель и т.д.).
3. Ответы сохраняются в **CollectedData** (по user_id и ключу).
4. При запросе к ИИ **сервис промптов** подставляет эти данные в **PromptTemplate** и формирует итоговый промпт.
5. Промпт уходит в LLM; ответ возвращается в чат и сохраняется в **Message**.

## 5. Пошаговый план разработки и интеграции

### Этап 0. Подготовка (1–2 дня)

| Шаг | Действие |
|-----|----------|
| 0.1 | Создать репозиторий, Next.js-проект (TypeScript, Tailwind, App Router). |
| 0.2 | Настроить Prisma, выбрать БД (PostgreSQL для продакшена или SQLite для старта). |
| 0.3 | Описать базовые модели: User, Session, Message. Миграции. |
| 0.4 | Базовая авторизация (email+пароль или OAuth; для Telegram позже — по telegram_id). |

**Результат:** проект поднимается, пользователь может залогиниться, в БД есть пользователи и сессии.

---

### Этап 1. Веб-чат: минимальный диалог (3–5 дней)

| Шаг | Действие |
|-----|----------|
| 1.1 | API: создание сессии чата, отправка сообщения, получение истории. |
| 1.2 | Фронтенд: экран чата (список сообщений, поле ввода, отправка). |
| 1.3 | Сохранение каждого сообщения (user/assistant) в Message. |
| 1.4 | Заглушка ответа (например, эхо или фиксированная фраза) без ИИ. |

**Результат:** пользователь видит диалог в веб-интерфейсе, сообщения сохраняются в БД.

---

### Этап 2. Сбор данных о пользователе (3–5 дней)

| Шаг | Действие |
|-----|----------|
| 2.1 | Модели: CollectedData (userId, key, value, source), Scenario (id, name, steps). |
| 2.2 | Сценарий: список шагов (тип: вопрос / выбор / свободный текст), порядок, куда писать ключ в CollectedData. |
| 2.3 | Логика в бэкенде: при старте сессии или по правилам проверять, каких данных не хватает; отдавать следующий вопрос из сценария. |
| 2.4 | Фронтенд: отображение вопроса от бота, форма (текст / выбор вариантов), отправка ответа; сохранение в CollectedData. |
| 2.5 | После ответа — переход к следующему вопросу или к обычному чату, когда сценарий пройден. |

**Результат:** бот задаёт вопросы по сценарию, ответы сохраняются в структурированном виде (CollectedData).

---

### Этап 3. Генерация промптов и подключение ИИ (4–7 дней)

| Шаг | Действие |
|-----|----------|
| 3.1 | Модель PromptTemplate: name, type (system/user), body с плейсхолдерами `{{field}}`. |
| 3.2 | Сервис: на вход — userId (или sessionId), идентификатор шаблона; загрузка CollectedData пользователя; подстановка в шаблон → итоговый промпт. |
| 3.3 | Интеграция с Ollama/OpenAI: вызов API с собранным системным промптом и историей сообщений (последние N сообщений). |
| 3.4 | В чате: при отправке сообщения пользователем — генерация промпта → запрос к LLM → сохранение ответа в Message и отдача на клиент. |
| 3.5 | Админка или конфиг: редактирование шаблонов промптов и привязка к сценарию/каналу (опционально). |

**Результат:** ответы в чате генерирует ИИ; промпт строится из шаблона и собранных данных пользователя.

---

### Этап 4. Улучшение веб-чата и UX (2–4 дня)

| Шаг | Действие |
|-----|----------|
| 4.1 | Индикация набора ответа (typing), обработка ошибок и повтор запросов. |
| 4.2 | Поддержка длинных ответов (стриминг от LLM, если API поддерживает). |
| 4.3 | История сессий: список прошлых чатов, открытие выбранной сессии. |
| 4.4 | Базовая аналитика: какие данные собраны, сколько сообщений (для отладки и улучшения сценариев). |

**Результат:** стабильный и удобный веб-чат с историей и управляемыми промптами.

---

### Этап 5. Интеграция с Telegram (5–8 дней)

| Шаг | Действие |
|-----|----------|
| 5.1 | Создать бота через @BotFather, получить токен. |
| 5.2 | Сервис Telegram: Telegraf или node-telegram-bot-api; приём сообщений, отправка ответов. |
| 5.3 | Идентификация пользователя: по telegram_id; создание/поиск User в БД (привязка telegram_id к User). |
| 5.4 | Один и тот же слой бизнес-логики: создание сессии (channel=telegram), сохранение Message, запуск сценария сбора, генерация промпта, вызов LLM. |
| 5.5 | Адаптация сценария под Telegram: короткие шаги, кнопки (reply keyboard / inline) где уместно. |
| 5.6 | Деплой: процесс Telegram-бота (PM2 или отдельный контейнер), доступ к той же БД и API (или к сервису промптов). |

**Результат:** пользователь может общаться с тем же логическим ботом и в вебе, и в Telegram; данные и промпты общие.

---

### Этап 6. Безопасность, политики, масштаб (по мере необходимости)

| Шаг | Действие |
|-----|----------|
| 6.1 | Ограничение частоты запросов (rate limit) на API и к LLM. |
| 6.2 | Согласие на сбор данных, политика конфиденциальности, хранение только необходимых полей. |
| 6.3 | Аудит логов (кто что спрашивал, без хранения тел сообщений в логах при необходимости). |
| 6.4 | Резервное копирование БД; мониторинг (здоровье API, ошибки LLM). |

---

## 6. Рекомендации по реализации

1. **Сначала веб, потом Telegram** — общая логика (сбор данных, промпты, вызов ИИ) одна; Telegram — ещё один канал ввода/вывода.
2. **Сервис промптов отдельно** — одна функция/модуль: (userId, templateId, опции) → готовый промпт. Так проще тестировать и подключать новые каналы.
3. **Сценарии в БД или конфиге** — хранить шаги сценария и привязку полей к CollectedData, чтобы менять вопросы без деплоя кода.
4. **Версионирование шаблонов** — при изменении промптов хранить версию или дату, чтобы можно было откатиться или A/B тестировать.
5. **Логирование запросов к LLM** — без тел сообщений по умолчанию; при необходимости — только с согласия и в защищённом хранилище.
